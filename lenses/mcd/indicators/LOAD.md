# MCD Lens: LOAD (Dynamic Code Loading) Indicators

> **Core MCD position:** Dynamic loading is where static analysis fails, the point where visible code diverges from what actually executes. In dependency code, dynamic loading of non-literal input is a primary mechanism for payload delivery.

## Severity Baselines

| Atom | Baseline Severity | Notes |
|---|---|---|
| `LOAD.EVAL` | High | Eval of non-literal input in dependency code |
| `LOAD.IMPORT` | Medium | Computed module names prevent static enumeration of loaded code |
| `LOAD.DYLIB` | Medium-High | Native library loading bypasses language-level controls |
| `LOAD.REFLECT` | Medium | Depends on what's being accessed and whether access controls are bypassed |
| `LOAD.DESER` | High | Deserialization of untrusted data with permissive deserializer |
| `LOAD.CODEGEN` | Medium | Depends on inputs and whether generated code is executable |
| `LOAD.WASM` | Medium-High | Depends on module source and import grants |
| `LOAD.MEMCHAIN` | High | Multi-stage in-memory execution has very limited benign justification in dependencies |

## Escalation Factors

- **Input is attacker-influenced.** The string/object passed to eval, deserializer, or import derives from network, user input, env vars, or external files. The single most determinative factor.
- **Preceded by XFRM activity.** Decoded/decrypted content immediately before loading is the canonical payload delivery chain.
- **Module name is computed, not literal.** For `LOAD.IMPORT`, static analysis can't determine what loads.
- **Deserialized objects use lifecycle hooks.** `__reduce__`, `readObject()`, `__wakeup`: the direct RCE path.
- **WASM module arrives over network at runtime.** Dynamic sourcing makes loaded code opaque to all prior analysis.
- **Library not in dependency manifest.** `dlopen()` targeting a path not in declared dependencies.
- **Reflection bypasses access controls.** `setAccessible(true)`, `__getattribute__` overrides.
- **Codegen target is executable memory or callable.** Generated bytecode/native code immediately invoked.
- **Error handling is suppressive.** Bare `try/except: pass` around LOAD calls. The author anticipated and masked failures.
- **LOAD site is in a dependency, not application code.** Supply chain risk where the author had deliberate embedding opportunity.

## De-escalation Factors

- **Input is a compile-time constant.** Literal string eval'd; verify no code path substitutes non-literal.
- **Strict allowlist gates loading.** Pre-validated module names, restricted unpickler, class whitelist.
- **LOAD site is sandboxed.** Subprocess with dropped privileges, seccomp filter, WASM with no host imports.
- **Idiomatic framework use.** Plugin registration, DI, ORM requiring reflection. Verify full data flow.
- **Serialized data integrity-verified.** HMAC/signature before deserialization, which doesn't eliminate internal producer risk.

> **Caveat:** Any de-escalation relying on "the input is safe" must be verified through complete data flow, not assumed from local context.

## Interpretive Combinations

| Combination | MCD Interpretation | Severity |
|---|---|---|
| `XFRM.ENCODE` + `LOAD.EVAL` | Encoded payload decoded and eval'd, canonical single-stage delivery | Critical |
| `NETW.HTTP` + `LOAD.EVAL` | Code fetched and executed, remote code execution | Critical |
| `NETW.HTTP` + `LOAD.DESER` | Serialized payload from network deserialized, RCE vector | Critical |
| `LOAD.IMPORT` + `FSYS.WRITE` | Module written then imported, two-stage write-load | High |
| `LOAD.DYLIB` + `XFRM.ENCODE` | Native library path obfuscated, conceals loaded binary | High |
| `LOAD.REFLECT` + `PRIV.*` | Reflection bypassing access controls for privilege operations | High |
| `LOAD.WASM` + `NETW.HTTP` | WASM fetched at runtime with host function imports | High |
| `LOAD.EVAL` + `EXEC.SHELL` | Eval constructs → shell execution, two-stage chain | Critical |
| `LOAD.MEMCHAIN` + `XFRM.ENCODE` | Multi-stage in-memory decode-execute chain | Critical |

## MCD-Specific Disambiguation

### LOAD.EVAL vs EXEC.SHELL: substrate matters
Through the MCD lens, both achieve arbitrary code execution but the blast radius differs. `LOAD.EVAL` executes within the language runtime with access to imported modules and in-process state. `EXEC.SHELL` has OS-level access. The two-stage combination (`eval` → `os.system`) gives both capabilities.

### LOAD.DESER: no explicit eval
Through the MCD lens, `LOAD.DESER` is as dangerous as `LOAD.EVAL` despite having no explicit `eval` call. The code execution is implicit in object reconstruction. Do not require evidence of eval syntax to escalate deserialization of untrusted data with a permissive deserializer.
